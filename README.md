# MVP Backoffice

Field service management and CRM designed for water-treatment businesses. Three PWAs (admin,
technician, customer) on top of a vanilla PHP/MySQL backend. 

Released under MIT with the Commons Clause restriction. You can read, fork,
modify, self-host, and contribute. You cannot sell it. See `LICENSE`.

## Stack

- PHP 8.1+ with PDO, no ORM
- MySQL / MariaDB
- Vanilla JS, HTML, CSS for all three front-ends (single-file PWAs)
- Apache `.htaccess` rewrites for routing
- Optional integrations: Gmail (service-account or OAuth), QuickBooks Online,
  FCM and APNS push

```
    |   +-- book.php        public lead-booking endpoint
    |   +-- upload_*.php    direct upload handlers (auth via JWT)
    +-- config/             DB, JWT, version
    +-- middleware/auth.php JWT middleware, sendJson / sendError helpers
    +-- routes/             route handlers, one per resource group
    +-- private/            service-account.json lives here (gitignored)
    +-- storage/signatures/ uploaded signatures (gitignored)
```

## Setup

1. Create a MySQL database and user.
2. Run `database.sql` against it. This creates all 37 tables (users,
   customers, equipment, appointments, invoices, payments, parts_catalog,
   push_subscriptions, qbo_* mapping, service_types, tax_rates, time
   tracking, etc.), the two reporting views, and seeds an admin /
   technician / customer demo account, all with the password `changeme`.
   Customer records, pricing, OAuth tokens, and push subscriptions from the
   source production dump have been stripped; only schema, indexes, FKs,
   and a small set of generic seed rows remain.
3. Copy `.env.example` to `.env` and fill in real values, or set the same
   variables in your Apache vhost / cPanel / Docker config. At minimum,
   `DB_*` and `JWT_SECRET` must be set.
4. Point a vhost at the repo root. The `api/public/.htaccess` does the
   rewrites; requests to `/api/...` get routed through `api/public/index.php`.
5. Visit `/admin.html` and sign in with `admin@example.com` / `changeme`.
   Rotate the password immediately.

## API URL configuration

The three PWAs each have a hardcoded `API_URL` at the top of the script
section pointing at `https://example.com/api/public`. Replace that with your
domain before deploying, or fork the load order so it reads from a JSON
config endpoint.

## Routes (overview)

```
POST   /auth/login                   issue access + refresh tokens
POST   /auth/refresh                 rotate access token
POST   /book                         public lead capture (no auth)
GET    /version                      current build string (no auth)
GET    /cron/reminders?secret=...    daily reminder job (no auth, secret)
GET    /gmail/callback               Gmail OAuth redirect target
GET    /qbo/callback                 QuickBooks OAuth redirect target

/admin/*       admin-only routes (customers, equipment, invoices,
               appointments, settings, gmail, qbo, parts, service types,
               time-tracking, weekly reports)
/tech/*        technician routes (own jobs, customers, invoices, notes,
               clock in/out, completed jobs)
/customer/*    customer portal routes (own appointments, invoices, notes,
               equipment, water tests, salt delivery requests)
```

Each resource group lives in `api/routes/{group}.php`. The front controller
`api/public/index.php` is the source of truth for which routes exist and who
can hit them.

## Conventions worth knowing

- No foreign-key constraints on new tables - referential integrity is
  enforced in PHP. The starter schema in `database.sql` does declare a few
  FKs on legacy tables; later migrations should not.
- `sendJson()` and `sendError()` (in `api/middleware/auth.php`) exit
  internally; callers should not run code after them.
- PDFs are generated by a pure-PHP generator in `api/routes/invoice_pdf.php`
  and its subclasses (`appointment_report_pdf.php`, `weekly_report_pdf.php`).
  No external library.
- QuickBooks Online: invoice and payment sync lives in `api/routes/qbo.php`.
  IDS query escaping uses `str_replace("'", "''", ...)` then `urlencode()`;
  do not pass the query through `parse_str` / `http_build_query`.

## Brand assets

`Prisma.otf` (used in the wordmark), `ntsky.jpg`, `blk_chrome.jpg`, and
`chrome.jpg` ship in the repo so a fresh checkout renders correctly. The
brand logo at `assets/logo.png` is intentionally not bundled; drop your own
in that path or set the invoice header to text-only by editing
`PDF_LOGO_PATH` in `api/routes/invoice_pdf.php`.

## What's out of scope here

- No automated tests.
- No CI config.
- No Docker setup. It's plain PHP; drop it on any LAMP host.
- No Composer dependencies.
- No financial tools, built in interconnect for Quickbooks Online to handle this.
